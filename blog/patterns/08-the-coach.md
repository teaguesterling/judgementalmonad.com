# The Coach

*Continuous observation with suggestions. The ratchet's observation phase, automated.*

---

## The problem

The agent works. The tools are available. The mode is set. But the agent doesn't always use what it has effectively. It searches for a function name with `file_search` when `find_definitions` would give it the answer with type information and scope. It reads files one at a time when `file_read_batch` would read all five in one call. It runs tests after every single edit when batching the edits and testing once would be more efficient.

The knowledge of "what to use when" currently lives in:
- The Strategy Instruction (pattern 2) — one-time, before the task
- The Mode Controller (pattern 7) — switches configurations on failure

Neither provides *ongoing* guidance. Neither says "hey, you just searched for `getattr` three times in different files — use `find_callers('getattr')` instead."

## The pattern

A hook that fires periodically (every N tool calls), analyzes recent tool usage, and injects a brief suggestion into the conversation. Not enforcement — guidance. The agent can ignore it.

### What the coach observes

Using Fledgling's conversation analytics:
- **Repeated patterns**: Same search three times → suggest a batch or semantic tool
- **Tool underuse**: `find_definitions` available but never called → suggest it when the agent is searching for function names
- **Inefficiency**: Five sequential `file_read` calls → suggest `file_read_batch`
- **Anti-patterns**: Editing without testing for 10 turns → suggest running tests
- **Context gaps**: Agent editing `evaluator.py` but hasn't read the test file that covers it → suggest reading the test

### What the coach says

Short, specific, actionable:
```
[coach] You've called file_search("getattr") on 3 files separately.
find_callers(function_name="getattr") searches all files at once
and shows the calling context.
```

```
[coach] You've edited src/evaluator.py 4 times without running tests.
Consider run_tests() to verify before continuing.
```

```
[coach] You're reading files one at a time. file_read_batch can
read multiple files in one call:
file_read_batch(["src/tokenizer.py", "src/parser.py", "src/evaluator.py"])
```

### What the coach doesn't do

- **Doesn't enforce.** The agent can ignore every suggestion.
- **Doesn't decide.** The coach observes and suggests, it doesn't change modes or tools.
- **Doesn't use trained judgment.** The suggestions are pattern-matched from the tool call log, not generated by an LLM.
- **Doesn't slow down.** Fires every N calls (not every call), brief output, no extra turns.

## The grade analysis

The coach is System 3* in Beer's VSM — the audit/monitoring function. It observes downward (watches the agent's tool calls) and reports upward (injects suggestions). It doesn't modify the controlled system's configuration — that's the Mode Controller's job. It adds *information* to the agent's context, not *constraints* to its tool set.

The coach is a W-axis intervention: it adds context (world coupling) that changes d_reachable. The suggestion "use find_callers instead of file_search" is information that activates different paths through the agent's decision surface. Same mechanism as the Strategy Instruction, but continuous and context-specific.

## Implementation

### Via Claude Code hooks

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "command": "python3 coach.py --log-dir .claude/ --every 5",
        "description": "Suggest more efficient tool usage every 5 calls"
      }
    ]
  }
}
```

The hook reads the last N tool calls from the conversation log, pattern-matches against known inefficiency patterns, and prints a suggestion if one applies. Claude Code includes the hook's output in the next turn's context.

### Via Fledgling

Fledgling already has conversation analytics (`ChatToolUsage`, `ChatSearch`). The coach queries these:

```sql
-- Detect repeated searches
SELECT tool_name, arguments->>'pattern' as pattern, count(*) as times
FROM recent_tool_calls(5)
WHERE tool_name = 'file_search'
GROUP BY pattern
HAVING count(*) >= 3;
```

If the query returns results, the coach suggests the semantic alternative.

### Via blq

For build/test patterns, the coach queries blq:

```sql
-- Detect edit-without-test pattern
SELECT count(*) as edits_since_last_test
FROM recent_tool_calls(20)
WHERE tool_name LIKE 'file_edit%'
AND sequence_number > (
    SELECT max(sequence_number) FROM recent_tool_calls(20)
    WHERE tool_name = 'run_tests'
);
```

## The ratchet connection

The coach IS the ratchet's observation phase, running continuously. Every suggestion is a data point: "the agent didn't know to use find_callers." Over time, these data points tell you:
- Which tools the agent underuses (need better descriptions or strategy instructions)
- Which patterns recur (candidates for combinators or new tools)
- Which suggestions the agent follows vs ignores (the followed ones become part of the strategy instruction; the ignored ones may be wrong)

The coach feeds the Quartermaster: "agents working on bug-fix tasks use file_search 3x more than find_callers, even when find_callers is available." The Quartermaster's next kit revision promotes find_callers to the default set and adds a strategy hint.

## The anti-pattern: the backseat driver

A coach that fires on every tool call, provides long explanations, or second-guesses the agent's decisions adds noise without value. The agent stops reading the suggestions. The output tokens increase. The cost goes up.

The coach should be:
- **Infrequent**: Every 5-10 calls, not every call
- **Brief**: One or two lines, not a paragraph
- **Specific**: Name the tool and the pattern, not general advice
- **Ignorable**: A suggestion, not a requirement
